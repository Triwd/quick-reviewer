# Meta Skill - 学术论文审查协调器

## 概述

本SKILL定义完整的论文审查工作流，协调其他各SKILL完成综合审查。作为"协调器"角色，负责任务规划、进度控制和结果整合。

## 详细执行步骤

### Step 1: 接收论文文件

确认用户上传的论文文件，记录文件信息：
- 文件名
- 文件类型（.doc/.docx/.pdf/.txt/.md）
- 预估大小

### Step 2: 评估文件大小

读取文件内容，评估总字符数：

```
if 总字符数 < 20,000:
    → 完整解析（Step 3b）
else:
    → 分片解析（Step 3a）
```

### Step 3a: 分片解析（大文件）

**分片策略**：

1. **按章节分片**（首选）
   - 根据章节标题（第X章、X.X节）进行分割
   - 每个章节作为一个分片
   - 保留章节标题在分片开头

2. **按段落分片**（章节过长时）
   - 当单章节 > 10,000字符时，按段落拆分
   - 确保边界在段落结束处

3. **重叠区域**
   - 相邻分片保留约500字符重叠
   - 用于保持上下文连贯性

### Step 3b: 完整解析（小文件）

一次性读取完整文档内容，直接进行后续审查。

### Step 4: 提取元数据

解析文档，提取以下信息：

| 元数据 | 提取方法 | 用途 |
|--------|---------|------|
| 中文标题 | 文档开头查找 | 报告头部 |
| 英文标题 | Abstract前查找 | 报告头部 |
| 中文摘要 | "摘要"关键字后 | 内容审查 |
| 英文摘要 | "Abstract"关键字后 | 内容审查 |
| 关键词 | "关键词/Key words"后 | 主题分析 |
| 章节结构 | 标题层级提取 | 结构审查 |
| 图表数量 | 扫描"图/表/Fig/Table" | 图表审查 |
| 参考文献数 | 扫描"[数字]" | 参考文献审查 |

### Step 5: 并行审查

可同时执行的审查维度（互不影响）：

**5.1 结构审查** → 读取 `structure_review.md`
- 检查章节完整性
- 检查标题层级
- 检查逻辑连贯性

**5.2 语法审查** → 读取 `grammar_spelling.md`
- 检查错别字
- 检查英文拼写
- 检查标点符号
- 检查格式规范

**5.3 图表审查** → 读取 `figures_tables.md`
- 检查编号连续性
- 检查中英文对应
- 检查引用准确性

### Step 6: 深度审查

按章节顺序执行，需保持上下文：

**6.1 引言与相关工作** → 读取 `logic_content.md`
- 评估引言逻辑
- 检查文献综述质量

**6.2 方法描述** → 读取 `logic_content.md`
- 检查问题定义清晰度
- 检查算法描述完整性
- 检查公式推导正确性

**6.3 实验设计** → 读取 `logic_content.md`
- 检查对比方法充分性
- 检查评价指标合理性
- 检查消融实验完整性

**6.4 结论与参考文献** → 读取 `references.md` + `logic_content.md`
- 检查结论质量
- 检查参考文献规范性
- 检查引用-文献匹配

### Step 7: 综合评审

读取 `reviewer_simulator.md`，从审稿人视角评估：
- 创新性（Novelty）
- 技术正确性（Technical Correctness）
- 实验充分性（Experimental Validation）
- 写作质量（Presentation Quality）
- 文献综述（Related Work）

给出推荐意见：Accept / Minor Revision / Major Revision / Reject

### Step 8: 整合报告

**数据处理**：
1. **去重**：合并各维度发现的相同问题
2. **分级**：按严重程度分类（Critical/Major/Minor/Suggestion）
3. **排序**：按严重程度和位置排序
4. **统计**：生成问题统计信息

**报告结构**：
```
1. 基本信息
2. 执行摘要（总体评分+推荐意见）
3. 各维度详细审查结果
4. 问题清单（分级列出，每条须包含：位置、类型、原文、修订文本）
5. 修改建议（每条须包含：位置、类型、原文、修订文本）
6. 总结
```

### Step 9: 保存报告

将报告保存到 `./report/` 目录：

```
./report/
└── review_report.md          # Markdown格式报告
```

可选保存（用户要求时）：
```
./report/
├── review_report.md          # 主报告
└── review_report.json        # 结构化数据
```

## 临时文件管理

### 文件生命周期

| 文件类型 | 生成时机 | 保留时间 | 最终处理 |
|---------|---------|---------|---------|
| 提取的原始文本 | 解析阶段 | 审查期间 | ❌ 删除 |
| 分片文件 | 解析阶段 | 审查期间 | ❌ 删除 |
| 元数据JSON | 解析阶段 | 审查期间 | ❌ 删除（可选保留）|
| 审查报告 | 整合阶段 | 永久 | ✅ 保留 |

### 清理时机

在以下情况完成后执行清理：

✅ **必须清理**（默认）：
- 所有审查维度已完成
- 审查报告已生成并保存到 `./report/`

❌ **暂缓清理**（用户明确要求时）：
- 用户要求保留中间文件供检查
- 需要分阶段多次审查

### 保留文件清单

```
./report/                      # 报告目录
└── review_report.md          # Markdown格式审查报告（必须保留）

# 以下文件可选保留（用户要求时）
./report/
├── review_report.md          # 主报告
├── review_report.json        # 结构化数据
└── chunks/                   # 分片文件（供用户检查）
    ├── chunk_0000.txt
    └── ...
```

### 问题清单格式

每条问题**必须**包含以下四个字段，以表格形式输出：

```markdown
## 问题清单

### 🔴 严重问题（Must Fix）

| # | 位置 | 问题类型 | 原文 | 修订建议 |
|---|------|---------|------|---------|
| 1 | 第3章 第2节 第1段 | 英文拼写 | "Critc网络" | "Critic网络" |
| 2 | 参考文献[5] | 格式错误 | "nature" | "Nature" |

### 🟠 中等问题（Should Fix）

| # | 位置 | 问题类型 | 原文 | 修订建议 |
|---|------|---------|------|---------|
| 3 | 第1章 第3段 | 引用格式 | "[3][4]" | "[3, 4]" |

### 🟡 轻微问题（Minor）

| # | 位置 | 问题类型 | 原文 | 修订建议 |
|---|------|---------|------|---------|
| 4 | 图4.2标题 | 英文格式 | "Fig 4.2" | "Fig. 4.2" |

### 🔵 建议项（Suggestion）

| # | 位置 | 问题类型 | 原文 | 修订建议 |
|---|------|---------|------|---------|
| 5 | 第5章 结论 | 内容补充 | "取得了良好效果" | "在简单场景中取得95.3%的避障成功率，在复杂场景中达到82.7%" |
```

**字段说明**：

| 字段 | 要求 | 示例 |
|------|------|------|
| **位置** | 精确到章节、段落或行。格式："第X章 第Y节 第Z段" | "第3章 3.2节 第1段" |
| **问题类型** | 使用统一分类：拼写错误、语法错误、标点符号、格式规范、引用格式、内容缺失、逻辑问题、术语不一致 | "英文拼写" |
| **原文** | 引用原始文本片段，用引号包裹。如文本较长可截取关键部分并用"..."省略 | "Critc网络" |
| **修订建议** | 给出修改后的具体文本，用引号包裹。对于内容缺失类问题，给出建议补充的文本 | "Critic网络" |

> **重要**：不要只写"建议修改"或"需要补充"等笼统描述，必须给出**可直接采用的具体修订文本**。

## 使用示例

### 示例1：标准全面审查

```markdown
用户：请帮我全面审查论文 manuscript.doc

大模型执行：
1. 读取 00-meta/SKILL.md（本文件）
2. 按"完整工作流"执行全部10个步骤
3. 输出：./report/review_report.md
4. 清理所有临时文件
```

### 示例2：保留中间文件

```markdown
用户：请审查论文，保留中间文件供我检查

大模型执行：
1-8. 同上（执行审查）
9. 保存报告 + 保存分片文件到 ./report/chunks/

```

### 示例3：多轮增量审查

```markdown
用户：我已修改了第3章，请重新审查这一章

3. 更新报告中的第3章部分
4. 保留其他章节结果不变
5. 清理临时文件
```

## 注意事项

### 1. 分片边界处理

- **避免截断**：不在句子中间分片
- **保留标记**：分片开头保留章节标题
- **引用完整**：确保引用标记[1]不被截断

### 2. 跨分片问题识别

以下问题需要全局视角：

| 问题类型 | 识别方法 | 记录方式 |
|---------|---------|---------|
| 术语一致性 | 维护全局术语表 | 统一记录 |
| 引用匹配 | 全局引用-文献映射 | 统一记录 |
| 图表编号 | 全局图表清单 | 统一记录 |
| 公式编号 | 全局公式清单 | 统一记录 |

### 3. 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 文件读取失败 | 提示用户检查文件格式 |
| 编码问题 | 尝试多种编码（UTF-8/GBK）|
| 解析超时 | 增加分片粒度，减小单分片大小 |
| 内存不足 | 强制分片，逐片处理 |
